Review

本周阅读了一篇文章《Modern Distributed Tracing Systems Architecture》

一个分布式系统拥有着许多互相沟通的服务，这样会带来很大的不可遇见的挑战。作为一个开发者，我不想也不愿意去debug或者是猜为什么，哪里，什么时候在生产环境发生了故障，而应该了解到更加详细的日志信息。这就是理想的分布式追踪系统应该提供给我们的功能，它从所有服务中提取信息，它也集成了查询，报警还有可视化的能力从而帮助我们意识到系统里面发生了什么。

比如Google的Dapper，类似的还有Pinpoint，Magpie & X-Trace，Dapper在将跟踪传输到跟踪系统的方式上提供了采样和应用程序级别的透明度。随着人们更加在意分布式追踪系统，这时标准的API还有文档就应该出现了。

这时开放式分布式API项目（Opentracing API Project）就来到了历史舞台前面。它提供了开放式分布式追踪系统的标准，OSS services & OSS packages不去绑定特定的供应商，也就是说，隔离地检测其代码，并通过一次配置更改即可更改跟踪技术。一个典型的现代分布式跟踪软件系统如下图所示，具有4个主要组件：

  • Tracing Instrumentation API (跟踪规范API(装饰应用程序代码))
  • Wire Protocol (有线协议(添加到HTTP / RPC请求中的元数据以及应用程序数据)
  • Data Protocol (数据协议(我们服务发送到跟踪/分析系统的数据))
  • Analysis System (分析系统((具有数据库和交互式UI的提取系统，它被用于处理跟踪数据))

首要因素

在每一个分布式的追踪系统中，最重要的就是span，这代表这一个独立的单元的工作在分布式系统中完成了。它是一个定时操作，代表系统每个组件贡献的一部分工作流程。Spans一般和其他spans有关联，很多歌spans就组装成了一个完整的”trace”。一个追踪可以看作是一个请求在系统中移动的生命周期。也就是说，一个spans的DAG.特定的span封装了操作名称，开始和结束时间戳，Tags＆Logs的key:value以及SpanContext。如下图所示，A是具有多个子跨度的父span，每个span的开始和结束时间在X轴上表示。

作为一个一个服务发送给其他服务的特定请求，带有跟踪内容信息的附加元数据是被传播了的。举一个例子来说，一个HTTP请求，它的headers带上了X-Trace或者Trace-Parent-ID是被添加上了的。

分布式追踪系统的类型

我们可以通过跟踪数据是怎样从宿主到跟踪系统传送的方式去区分不同的追踪系统。

直接追踪


Zipkin是一种分布式的追踪系统，它支持Http，Kafka还有Scribe同传送给collector，其中collector负责手机，处理，存储这些信息。它的存储和Cassandra还有ElasticSearch是兼容的，并且Zipkin还有着可以通过UI提供查询的支持。

基于代理的传输


Uber开发的Jaeger并且开源了它(被Dapper和Zipkin启发的)，因此，Jaeger可以看做是在Zipkin改动了一些架构，增加了一些新的功能。一个主要的改动就是存在一个网络代理jaeger-agent，它需要部署在所有的宿主中作为一个基础架构的代理。它从客户端抽象了路由与发现jaeger-collector的功能。它同样提供控制流以便于更加有效地传输跟踪，并具有对来自本地主机的传入UDP跟踪的缓冲支持，同时根据基础结构确定理想的采样策略。

其中New-Relic,Instana & AWS X-Ray有同样的效果。

基于本地/数据中心卫星的分析和传输


Google Dapper的合伙人创造了Lightstep，它是一种独一无二的绩效管理工具，可分析100％的未采样生产痕迹。它们的做法是部署一个卫星到宿主的数据中心或者是云端上。这一些卫星分析来自服务，web,手机端&无服务功能的数据，从而通过历史规范定位数据，用户定义的阀值还有相关的指标。这一些数据送往SAAS平台，SAAS平台是存储着一些数据，并且提供可视化UI操作的地方。一旦每一个数据都被分析到了，那么捕获百万分之一的异常是可能的。

结论

企业可以根据自身的情况选择不同的分布式追踪系统，这通常取决于系统的任务关键程度。像Lyft，它可以保证100%的请求追踪结果，而Google允许一定的误差，更注重于未来的可扩展性，于是使用了Dapper。
