Review

本周阅读了一篇Medium上的文章《Understanding Apache Kafka — The messaging technology for modern applications》

作者在文章中简单的介绍了一下Kafka的作用以及Kafka内部的设计架构。

Kafka简介

Apache Kafka已经成了建造一款可靠的大容量消息系统事实上的标准。然而比较特别的是，Kafka内部简单的架构让它可以很简单地可以被企业级的架构整合进去。在一个宏观的角度看待，它是一个消息中心，它把生产者和消费者很好的解耦，同事它也提供了可靠的大规模消息分发给消费者的机制。Kafka有一个消息生产者，它会把消息传递给一个叫做topic的地方，这个topic会把消息以一种可靠的方式分发给消费者。它可以有几种不同类型的生产者还有消费者，这取决于使用的情况。

架构


Kafka的架构是很简单但很有效果的架构，它可以和大部分架构结合地很好。

Kafka在跨越了多个机器、多个数据中心，多个区域的集群节点上工作。Kafka有能力通过地理上的分布式资源扩展。一旦Kafka集群部署完毕了，外部的应用和系统可以通过4中标准APIs和集群交互。

生产者(Publisher) API

这个用于事件源头大规模地发送事件给KafKa集群。Kafka把事件集群存储在topic中，topic被分区存储了因此topic可以水平扩展而不是仅限于单个服务器上。这些分区通过大量节点复制，以防止其中某一个节点宕机之后数据丢失。在发布事件期间，生产者可以指定存储在topic中事件的分区。

消费者(Consumer) API

存储在topic中的事件分发给订阅了这些topic的消费者。和一般的基于topic的发布/订阅不一样，消费者可以完全控制从topic中阅读消息。消费者可以控制offset value这意味着可以指定消息的位置比如哪一个分区，从而消费者可以在制定的topic和分区中任意的一个位置读取消息。

此外，Kafka有着一个消费者群体的概念，因此具有相同竞争条件的消费者可以通过一个基于订阅的队列来实现负载均衡。这里可能存在很多消费者群体都订阅者一个制定的topic，而每一个消费者群体都会在同一时间收到一个消息的拷贝。消费者群体内给定的消费者受限于主题内给定的分区，因此一个群体内的消费者的数量不会大于主题内分区的数量。

连接器(Connector) API

有时候Kafka需要集成到已经存在的应用和系统。举一个例子来说，如果我们使用Kafka读取一些日志文件，然后使用一个流处理器去处理这些数据，Kafka提供了连接器API去实现一些可重用的组件叫做连接器。同一个连接器可以被同样使用Kafka文件集成的系统使用。

流(Streaming) API

存储在Kafka中的事件有时候需要实时处理比如说转型，相关性，聚合等。这些功能可以通过一个工作在连续记录流上的流处理器实现。Kafka流API可以把topic作为流事件的接口暴露出去，因此其他流处理引擎可以消费它并且处理它。

Kafka的优势

  • 建造云扩展，容错处理的消息处理平台的能力
  • 消息持久化和耐久力可以配置一个保留期时间段，在消费者一端允许消费者如果它们需要的话可以自由的处理消息。
  • 在分区内消息的顺序是有保证的，这在某一些关键场合中非常有用
  • 复制品允许消息可以被可靠的存储，因此在节点数为N个的集群中，就算有N-1个节点失效了，它也不会影响消息正常的分发给消费者。
  • 集群的扩展性允许通过不同的数据中心发布系统，这样可以获得最高级别的可用性和更好的性能。
  • 流API允许事件被流处理引擎实时地处理
