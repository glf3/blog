线程安全

并行计算的起源

摩尔定律的失效

摩尔定律：集成电路上可容纳的电晶体（晶体管）数目，每隔约24个月便会增加一倍。

　　然而在2004年，Intel宣布将4GHz的芯片的发布推迟到2005年，在2004年秋季，Intel宣布彻底取消4GHz计划。

多核CPU的兴起

　　从2005年开始，我们已经不在追求单核的计算速度，而着迷于研究如何将多个独立的计算单元整合到单独的CPU中，也就是我们常说的多核CPU。短短十几年的发展，专业服务器的CPU可能达到几百个。令人激动的是，摩尔定律在另外一个侧面又生效了。

　　顶级的计算机科学家唐纳德克努斯(Donald Ervin Knuth)评价这种情况：在我看来，这种现象的（并发）或多或少是由于硬件设计者已经无计可施了导致的，他们将摩尔定律失效的责任推脱给了软件开发者。

　　简化的硬件设计方案必然带来了软件设计的复杂性。换句话说，软件工程师正在完成硬件工程师无法完成的工作职责。

　　所以，如何让多个CPU有效并且正确地工作也就成为了一门技术，甚至很大的学问。多线程间如何保证线程安全，如何正确理解线程间的无序性、有效性、可见性，如何尽可能的提高并行程序的设计，又如何将串行设计的程序改造为并行程序。

　　因此，当我们使用，就引出了我们现在的命题，线程安全。

多线程的目的

为什么要使用多线程？可以简单的分两个方面来说：

  • 更好利用多个CPU核心的能力
  • 面对复杂的业务模型，并行程序更容易适应业务需求，也更容易模拟我们的现实世界。比如IO操作阻塞的时候，是不需要CPU参与的，这时候CPU就可以另开一个线程去做别的事情，等待IO操作完成再回到之前的线程继续执行即可。

必须了解的几个概念

相关概念

  1. 同步和异步的区别：
    1.1 同步方法调用一旦开始，调用者必须等到方法调用返回后，才能继续后续的行为。
1.2 异步方法更像是一个消息传递，一旦开始，方法调用就会立即返回，调用者可以继续执行后续的操作。异步方法通常会在另外一个线程中“真实”地执行。
  2. 并发和并行的区别：
    2.1 并发偏重于多个任务交替执行，而多个任务之间可能还是串行的。对于单核CPU来说，任务的执行都是并发交替串行的。
2.2 严格意义上来说，并行是真正意义上的同时执行。对于多核CPU来说，任务很可能是同时执行的。
  3. 临界区：
    临界区表示一种公共资源或是共享数据，可以被多个线程使用。
  4. 阻塞、非阻塞的区别：阻塞和非阻塞通常用来描述多线程之间的相互影响。
    4.1　阻塞：一个线程占领了临界区资源，那么其他所有需要这个资源的线程就必须在这个临界区中进行等待。
4.2 非阻塞：它与阻塞相反，它强调没有一个线程可以妨碍其他线程的执行。
  5. 死锁、饥饿、活锁：
    5.1 死锁：指的是线程之间互相持有对方需要的临界区资源，并且相互等待的例子。
5.2 饥饿：某一个或者多个线程因为种种原因无法获得所需要的资源导致一直无法执行。
5.3 活锁：多个线程主动释放资源和其他线程，那么就会出现资源不断在多个线程之间跳动，而没有一个线程可以同时拿到所有资源正常执行。

并发级别

　　由于临界区的存在，多线程之间的并发必须受到控制。根据控制并发的策略，我们可以把并发的级别进行分类。大致可以分为阻塞、无障碍、无锁、无等待。

  1. 阻塞：
    一个线程是阻塞的那么其他线程释放资源之前，当前线程无法继续执行。当我们使用synchronized或者重入锁时，得到的就是阻塞的线程。
  2. 无饥饿：
    如果线程之间是有优先级的，那么线程调度的时候总会倾向于满足高优先级的线程。也就是说，对于同一个资源的分配，是不公平的。对于非公平锁来说，系统允许高优先级的线程插队，这样可能会导致低优先级的线程产生饥饿，如果锁是公平的话，那么饥饿就不会产生。
  3. 无障碍：
    无障碍是最弱的一种非阻塞调度。两个线程如果是无障碍的执行，那么他们不会因为临界区的问题导致一方被挂起。换言之，大家可以一起进入临界区修改数据。对于无障碍来言，一旦检测到了共享数据被损坏，那么立即会对自己所做的修改进行回滚。
  4. 无锁
    无锁的并行是无障碍的。在无锁的情况下，所有的线程都能尝试对临界区进行访问，但不同的是，无锁的并行保证必然有一个线程能够在有限步内完成操作离开临界区。无锁的一个特点是可能会包含一个无限循环。在这个循环中，线程会不断地尝试修改共享变量。

Java中的线程安全

　　这里以Java作为例子，探讨Java语言中的并行计算和线程安全。

JMM的概念(Java内存模型)

　　我们需要在深入了解并行机制的情况下，再定义一种规则，保证多个线程间可以有效地、正确地协同工作。而JMM也就是为此而生的。

　　JMM的关键技术点都是围绕着多线程的原子性、可见性、和有序性来建立的。

  1. 原子性：指的是一个操作是不可中断的
  2. 可见性：可见性指的是但一个线程修改了某一个共享变量的值，其他线程是否能够立即知道这个修改。
  3. 有序性：指的是程序在执行时，可能会出现指令重排，重排后的指令与原指令顺序未必是一致的。

JVM（Java虚拟机）内存区域划分

　　
根据各个区域的定义，我们可以知道：

  • 类变量：保存在“方法区”
  • 实例变量：保存在“堆”
  • 局部变量：保存在 “虚拟机栈”

　　“方法区”和“堆”都属于线程共享数据区，“虚拟机栈”属于线程私有数据区。因此，局部变量是不能多个线程共享的，而类变量和实例变量是可以多个线程共享的。事实上，在Java中，多线程间进行通信的唯一途径就是通过类变量和实例变量。

　　也就是说，如果一段多线程程序中如果没有类变量和实例变量，那么这段多线程程序就一定是线程安全的。而对于类变量，实例变量，如何保证它们的共享和安全问题就成了我们现在需要考虑的。

JVM中内存区域模型


　　在JVM内部使用的java内存模型(JMM)将线程堆栈和堆之间的内存分开，要理解JMM的话，可以类比计算机硬件访问内存的模型。由于计算机的CPU计算速度和内存IO的速度有几个数量级的差距，因此，现在计算机都使用了高速缓存作为缓冲：将内存中需要使用的数据先复制到缓存，当运算结束之后再同步回内存。

　　因为JVM要实现跨硬件平台，因此JVM定义了自己的内存模型，但是因为JVM的内存模型最终还是要映射到硬件上，因此JVM内存模型几乎与硬件的模型一样：

每个Java线程都有一份自己的工作内存，线程访问变量的时候，不能直接访问主内存中的变量，而是先把主内存的变量复制到自己的工作内存，然后操作自己工作内存里的变量，最后再同步给主内存。

　　由于上述JMM的设计，当多个线程多个线程同时操作一个共享数据的时候，是可能出错的。为了避免这种情况的出现，这个时候就需要用到同步这一个技术。

同步(synchronized)就是：在多个线程并发访问共享数据的时候，保证共享数据在同一个时刻只被一个线程使用。

Java中的同步技术

加锁

　　为了保证共享数据在同一时刻只被一个线程使用，我们有一种很简单的实现思想，就是在共享数据里保存一个锁，当没有线程访问时，锁是空的，当有第一个线程访问时，就在锁里保存这个线程的标识并允许这个线程访问共享数据。在当前线程释放共享数据之前，如果再有其他线程想要访问共享数据，就要等待锁释放。

加锁的技术

  1. synchronized同步机制：
    synchronized是JDK支持的关键字，它的作用是实现线程间的同步。它的工作是对同步的代码加锁，使得每一次，只能有一个线程进入同步块，从而保证线程间的安全性。
  2. 可重入锁（ReentrantLock）：
    重复进入：可重入锁是可以反复进入的。当然这里的反复仅仅局限于一个线程。
中断响应：如果一个线程正在等待锁，它可以收到一个通知，可以无须等待，继续工作了。
  3. ReadWriteLock读写锁：
    读写分离的锁可以有效地帮助减少锁竞争，以提升系统性能。

锁的优化

  • 减小锁持有时间
  • 减少锁粒度
  • 锁分离
  • 锁粗化
  • Java虚拟机对锁的优化：
    1. 锁偏向
2. 轻量级锁
3. 自旋锁
4. 锁消除

无锁

　　对于并发控制而言，锁是一种悲观的策略。它总是假设每一次的临界区操作会冲突。

　　而无锁的话则是一种乐观的策略，它会假设对资源的访问是没有冲突的。既然没有冲突，自然不需要等待，所以所有的线程可以在不停顿的状态下继续执行。那遇到冲突怎么办呢？无锁使用一种策略叫做比较交换的技术(CAS)，具体无锁的概念和实现可以看我上周写的ARTS的Share。

线程安全代码对性能的影响

　　我觉得这个得分情况讨论

  • 单核CPU：
    如果执行多线程程序的话，因为多线程的执行永远是串行的，因此同步代码块不会对性能产生什么影响，这种情况下对性能影响比较大的是多线程频繁调度产生的消耗。
  • 多核CPU：
    多核CPU性能消耗主要是在锁的争夺上、线程调度上，比如对于synchronized，它底层的锁实现是非公平的，因此可能存在一个线程永远获取不到需要的锁的情况。因此，锁的争夺，以及线程的挂起，死锁，饥饿以及多线程调度都成了多核CPU上对于性能影响的一部分了。

JDK1.8下的ConcurrentHashMap

　　花了一点时间阅读ConcurrentHashMap的源码，大概知道了ConcurrentHashMap的基本工作流程和怎么实现的插入(put)和删除(remove)。

ConcurrentHashMap里面的基本数据结构

  • transient volatile Node< K,V >[] table;是一个数组，第一次插入数据的时候初始化，大小是2的幂次方。这就是我们所说的底层结构：“数组+链表（或树）”
  • Node< K,V >：用来保存Key-Value数据的基本数据结构
  • TreeNode< K,V >：继承自Node，用于维护树节点之间的关系，是TreeBin维护的数据结构
  • TreeBin< K, V >：维护红黑树的数据结构，内部维护了TreeNode以及操作它们的方法。

工作原理

　　ConcurrentHashMap内部维护了一个table，其中table[i]指向的一个链表或者是红黑树的数据结构。其中i的值代表的是Key经过hash运算之后的下标。

  • 插入：

    1. 当插入Node的时候，首先会针对Key做一个hash的操作，找到对应的table[i]。
2. 如果这时table不存在，那么便会创造table。如果table[i]不存在那么便会把Node插入table[i]。
3. 如果存在说明有冲突，当有冲突的时候，便会把节点插入到table[i]里面。
    3.1 如果当前table[i]是链表：当前链表长度大于一个阀值，便会把链表转换成红黑树。
    3.2 如果当前table[i]是红黑树：正常插入

  • 删除:

    1. 当插入Node的时候，首先会针对Key做一个hash的操作，找到对应的table[i]。
2. 如果这时table或者table[i]不存在，那么不会做任何事情。
3. 如果存在则会删除：
    3.1 如果当前table[i]是链表：正常删除
    3.2 如果当前table[i]是红黑树：那么当树节点数量小于阀值的时候，会把红黑树变为链表。

  • 保证线程安全的方式

    1. synchronized关键字：通过key获取到table[i]的值，以table[i]作为实例加锁，用到了synchronized(table[i])table只有一个线程访问。
2. Java中的Unsafe类用于CAS操作：Unsafe类是JVM开了一个后门，专门用于调用底层操作系统的方法，它提供了硬件级别的原子操作。比如创建table的时候，扩容的时候都用到了这个类。
