Algorithm

本周算法题329. 矩阵中的最长递增路径

题意：

分析：

  • 对于矩阵中的一个点来说，如果它比上下左右四个方向的点的值要小，那么我们连一条边，由于相等的数之间没有边，因此，这是一个有向无环图(DAG)。于是把问题转换成了有向无环图上的最长路径。
  • 因为最终入度为0的点是确定的，因此每一个节点到入度为0的点的距离也是确定的。
  • 因此，我们从入度为0的点开始逆推，可以做如下动态规划方程。dp(i) = max{dp(i), dp(j)+1}。(0 < i,j < n)
    1. 对于节点i来说，所有状态是dp(j)，代表所有从i出发能够到达的点j
    2. 状态之间的联系是dp(j) -> dp(i)转移，表达的意思是从i出发的最长路径(d[i])等于从i能到达的点的最长路径(d[j]) + 1，即dp[i] = dp[j] + 1 {0 < j < n}
    3. 转移状态的决策是max(选取最大值)
    4. 对于d(j)来说，d(j)代表的意思是从j出发能够到达的最长路径，因此对于d(i)来说，子问题也是最优解，对于这个状态转移方程而言，最优子问题，状态转移，状态转移决策都满足条件了

Review

本周阅读了一篇Medium上的文章《What is DevOps?》

  • DevOps（Development和Operations的组合词）是一组过程、方法与系统的统称，用于促进开发（应用程序/软件工程）、技术运营和质量保障（QA）部门之间的沟通、协作与整合。它是一种重视“软件开发人员（Dev）”和“IT运维技术人员（Ops）”之间沟通合作的文化、运动或惯例。透过自动化“软件交付”和“架构变更”的流程，来使得构建、测试、发布软件能够更加地快捷、频繁和可靠。它的出现是由于软件行业日益清晰地认识到：为了按时交付软件产品和服务，开发和运营工作必须紧密合作。

  • 当人们谈到“DevOps”时，他们并不一定意味着个人工具或工作头衔，而是更多的一系列实践，这些实践在开发人员和运营商中都是如此，以便将软件交付到最后用户以更全面，更有效的方式。这更像是一种敏捷的理念，试图打破组织孤岛，促进跨职能的团队合作。

  • 它是一个流行语来引用关于如何运行系统和工程组织的一系列模糊互联的概念，但不可否认的是近年来，运营团队的职责发生了很大的变化甚至趋同。越来越多的公司要求开发人员如果它们写的服务除了问题需要随叫随到，并且把Ops/SRE拉进来一起讨论，让Ops/SRE团队承担除了咨询之外更多的角色。

Tips

本周复习了Android WindowManager相关的知识。

Window、WindowManager、WMS

Window我们应该很熟悉，它是一个抽象类，具体的实现类为PhoneWindow，它对View进行管理。WindowManager是一个接口类，继承自ViewManager，WindowManager会将具体的工作交由WMS来处理，WindowManager和WMS通过Binder进行通信，WMS作为系统服务很多API是不会暴露给WindowManager的，这一点与ActivityManager与AMS很像。

Window、WindowManager、WMS可以用下图来表示。

WindowManager的关联类

在Activity的启动中会调用ActivityThread的performLaunchActivity方法，performLaunchActivity方法中回调用Activity的attach方法，PhoneWindow就是在Activity的attach方法中创建的。

从图可以看出，PhoneWindow继承自Window，Window通过setWindowManager与WindowManager发生关联。WindowManager继承自接口ViewManager，WindowManagerImpl是WindowManager接口的实现类，但是具体的功能都会通过桥接模式委托给WindowManagerGlobal去实现。

Window的操作

窗口添加的操作分为两部分，一部分是WindowManager处理部分，另一部分是WMS处理部分。Window分为三大类：Application Window、Sub Window、System Window，对于不同类型的窗口添加过程会有所不同，但是对于WMS处理部分，添加的过程基本是一致的，如图所示：

窗口的添加过程

我们从WindowManagerImpl.addView函数出发进行分析。

WindowManager.addView最终会调用到WindowManagerGlobal.addView，其中WindowGlobal.addView的过程中，创建了ViewRootImpl，最终会调用到ViewRootImpl.setView方法，ViewRootImpl有很多职责，主要有以下几点：

  • View树的根并管理View树
  • 触发View的测量、布局和绘制
  • 输入事件的中转站
  • 管理Surface
  • 负责与WMS进行通信

在setView中有很多逻辑，主要是调用了mWindowSession.addToDisplay方法，mWindowSession是IWindowSession类型的，它是一个Binder对象，用于进程间通信，IWindowSession是Client端的代理，它的Server是Session，如下图所示。

Session为什么会包含在WMS中了？mWindowSession.addToDisplay调用了WMS.addWindow方法，将自身作为参数传递给了WMS，这样每一个进程都会对应一个Session，WMS会用ArrayList来保存这个Session。

那么mWindowSession.addToDisplay剩下的工作就交给WMS来处理，在WMS中为这个添加的窗口分配Surface，并确定窗口次序，由此可见负责显示界面的是Surface，而不是窗口本身。WMS会将它所管理的Surface交由SurfaceFlinger处理，SurfaceFlinger会将这些Surface混合并绘制到屏幕上。

WMS添加窗口的部分以后会进行分析。

Share

分享一篇马蜂窝机票订单系统背后的设计原理以及实现。《状态机在马蜂窝机票订单交易系统中的应用与优化实践》

总结：

MSM (MFW State Machine)的定义和声明在Squirrel-Foundation的基础之上，抽取出Action概念，并对Action类配置起始状态、目标状态、触发的事件、上下文定义等，使得MSM可以根据Action类的配置和声明，来动态地生成Squirrel-Foundation的状态机定义，而不需要使用者再去定义一次状态机，操作更加简单，维护起来也更加容易。

状态机的使用场景不仅仅局限于订单交易系统，其他一些涉及到状态变更的复杂流程的系统也同样适用。
