ARTS第九周

　　ARTS，即：每周完成一个ARTS：每周至少做一个 leetcode 的算法题、阅读并点评至少一篇英文技术文章、学习至少一个技术技巧、分享一篇有观点和思考的技术文章，至少坚持一年。（也就是 Algorithm、Review、Tip、Share 简称ARTS）。

Algorithm

本周算法题239. 滑动窗口最大值

题意：

分析：

  • 求滑动窗口的极值的话可以用单调队列来解决

单调队列是这样一种数据结构：

  1. 一个这里队列中存储的是窗口最大值的下标。
  2. 如果队列为空，那么就将当前元素的下标添加到队列中
  3. 如果队列不为空，那么就有两种情况：
    1. 如果当前元素比队尾元素所指向的元素还要大，那么当这个元素进入到滑动窗口，队列尾那些比他小的元素所指向的元素，一定不可能是窗口中的最大值，所以就将队尾这些元素删去，直到队尾元素指向的元素比当前的元素大或者相等，然后将当前元素的下标加入队列。
    2. 如果当前元素比队尾元素所指向的元素要小或者相等，那么就直接将当前元素的下标加入到队列尾。
  4. 取的时候，只需要取队列头的元素就可以了，因为那就代表着当前窗口的最大值。取的时候要判断一下：
    1. 如果队头元素已经是窗口之外了，那么就要将队头出队。
    2. 否则直接取队头元素就可以了，它就是窗口中最大值对应的下标。

Review

本周阅读了一篇Medium上的文章《Creating APIs for the human developer》

作者介绍了一下编写API时需要注意一些最佳实践。

什么才是好的API

  • 完成了它的工作
  • 它能够很简单与便于使用
  • 它能够很安全的使用

完成了它的工作

这是API存在的意义和价值，它聚焦与问题并且解决了它。

它很容易理解并且便于使用

一个好的API是依据使用它的客户的现有的知识去打造的。如果需要做到便于使用的话，它需要遵守以下规则：

  • 尊重它所在的平台：比如Javascript是用驼峰原则书写的，那么如果你编写一个Javascript的工具的话也得遵守这个原则。
  • 遵守一定的规则
  • 不要提供太多的选项：保持API的简单是很有必要的，如果它需要扩展的话那么就扩展，如果不需要的话那么就去掉。不提供冗余的功能可以让你的用户更快、更容易的使用好它。

能够安全的使用

你的API一定不能给用户引入BUG，因此需要遵循一下规范。

  • 当操作有危险时，显示的声明它：有时候你直接了当的声明这个API可能造成的错误可以帮助用户在编写代码时发现可能意外的情况。
  • 需要友好地执行：永远不要去拿到不属于自己的资源，因为你不能保证没有人在用它们。
  • 智能的捕获错误：当你捕获错误的时候，需要写一份文档把对应的错误一一列举错来，这样用户可以很方便的发现出现了什么样的错误。
  • 防止继承：这是为了防止出现内部没有声明的接口被暴露出来，引起不必要的麻烦。

如何设计一个好的API

组织自己的哲学

应该知道主要是解决什么样的问题，完全理解它，不要引入冗余的功能。

做一些核心的决定

比如说是否使用Promises？这是同步还是异步的接口？是否组织用户的数据或者是返回一个新的对象？是否使用类还是普通功能？静态变量还是临时变量？

管理接口的版本和使用弃用的策略

有时候当需要弃用一个老旧接口的时候，最佳的策略并不是删除所有的接口，而是在接口后面加一个flag比如–legacy或者–insecure这样显式的声明这个接口已经被弃用。

原型

永远不要让实现泄露了接口设计里面内容

如果让用户通过接口名称知道了接口设计里面的内容的话，当你下一版重构需要重新改写功能的时候会很麻烦，因为你会发现之前的名字与它实际干的工作并不是一致的。

把文档写详细

很多时候用户并不会去翻阅文档，只有当下面情况发生的时候才会。

  • 他们需要拷贝一份快速使用导航
  • 当有一部分功能没生效的时候

当你的文档写的不是特别好的时候，不管你的接口怎样出色的完成的它的工作，用户永远不知道发生了什么。同样，发生错误的代码和信息也得写清楚。这样会帮助用户定位他们的问题。

总结

设计接口的话永远是一个不断权衡的过程：提供功能和防止它们做出损害自己之间做权衡。

找到平衡并不总是那么容易，你永远不能取悦每个人。只要记得你实际上正在为其他人构建工具，那些恰好是软件开发人员的人，那么你就正朝着正确的方向前进。

Tips

之前的项目有很成熟的MVP框架，但商城中是没有现成的框架的，这次在写商城短视频项目的时候简单的写了一个MVP的框架，总结一下。

MVP架构设计

无论是MVP还是MVVM。架构的设计都是为了对付后期不断变化的需求还有减小后期的维护成本和解耦来做的。

View

可以是Fragment或者是Activity，View中仅仅只做与展示相关的工作，它实现了View -> Presenter接口，它通过使用Presenter来获取数据，因为实现了让Presenter可以很方便的调它的接口来实现回调数据。

Presenter

主要的业务逻辑其实都放在这里面，比如处理数据和网络请求，还有一些脏活，累活都是它来做。它实现了Presenter -> Model的接口，这样当Model有数据回调时可以告诉它。

Model

具体获取数据的实现，可以是网络请求也可以是数据库操作，总之与数据打交道都是它来处理。它实现了Model -> Presenter接口。
