ARTS第十六周

　　ARTS，即：每周完成一个ARTS：每周至少做一个 leetcode 的算法题、阅读并点评至少一篇英文技术文章、学习至少一个技术技巧、分享一篇有观点和思考的技术文章，至少坚持一年。（也就是 Algorithm、Review、Tip、Share 简称ARTS）。

Algorithm

本周算法题235. 二叉搜索树的最近公共祖先

题意：


分析：

没有想到很巧妙的方法，在纸上模拟了几遍，发现对于节点p、q(p.val < q.val)来说，如果一个节点root满足：root.val >= p.val && root.val <= q.val，那么在搜索二叉树中，满足条件的第一个节点就是p和q的最近公共祖先。

  • 通过递归可以找到第一个满足条件的节点
  • 递归的边界是：
    1. root.val >= p.val && root.val <= q.val：说明找到了结果，直接返回
    2. root == null：节点为空，直接返回。

Review

本周阅读了一篇Medium上的文章《How to Become a Better Software Developer》

作者在文中给了如何成为一个更好的开发者的一些建议，如下所示：

  • 理解端到端
    1. 一个完整的软件开发的生命周期如图所示：
    2. 通过向每个人学习，您将能够预测他们的需求，缩短反馈循环并实现更频繁的交付。此外，它将赢得所有其他人的爱和尊重。
  • 明白客户的需求
  • 为工作选择正确的工具
  • 进行必要的安全实验
    1. 如明确要先测试哪一部分。
    2. 搭建一个小型的工作原型进行测试。
  • 站在巨人的肩膀上
    1. 如果开源库解决了当前遇到的一些问题的话，那么就用开源的吧，没必要去重复造轮子了。
  • 尝试重新实现
    1. 在生产项目中重新实现自己的框架是一件很愚蠢的事，但是如果是作为练习的话价值是非常高的。
    2. 学习开源项目的话可以是你可以从其他非常有经验的工程师中学习到很多。
  • 研究你的工作方式
  • 专注学习基础知识
    1. IT行业是一个快节奏的行业，但是它的基础知识是不会有大的变化的，因此，专注于基础知识，打好基础吧。
    2. 花一些时间熟悉经典的关于软件领域中一些公共设计模式的图书。
    3. 不用关注所有新鲜的事物，这样你会陷入FOMO，相反，要留意社区中已经存在一段时间的事情，并且已经超越了初步讨论的秒作。
  • 分享知识
    1. 通过博客的形式，或者把一部分工作委托给他们，然后用这种方式去做更好的事情。
  • 不要责备自己和他人
    1. 当事故发生的之后不要去责备他人，而是要积极解决，并且在事后一起讨论导致事故发生的原因以及为什么会发生，以便于提升系统还有工作的方式。
  • 不要做一个混蛋
    1. 在开源社区一些人另一方面他们会哄骗新想法，不尊重新人，并向周围的每个人展示粗鲁的行为。不要做这样的人，而是要尝试着去开源社区贡献代码，发表演讲以及贡献文章。
  • 把以上的记住
    1. 关于我们工作的最好一点是没有限制，总有许多方法去屠龙，不管你是初学者还是专家，把这些记住，它们应该帮助你找到自己的方式，并在每个步骤中成为更好的开发人员。
