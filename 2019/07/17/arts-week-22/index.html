Review

本周阅读了一篇Medium上的文章《SOAP vs REST.》

作者在文章简略的说明了SOAP和REST的区别以及在什么情况下使用它们。

简介

SOAP是Simple Object Access Protocol的首字母缩略，在Service Oriented Architecture(SOA，面向服务的软件架构)中已经多年占据了主导焦点。它成为了应用互通性之间最受欢迎的协议。

然而，它的扩展被延迟了，因为出现了第二种可以替代它的分布式计算方案叫做REST(Representational State Transfer，表现层状态转换)。REST自从HTTP协议问世以来(1991年)就存在了，在过去的十年里，它已经发挥了很大的作用，被用于近70％的公共API中。

但是哪一个更加好呢？我应该用哪一种协议？为了回答这些问题，你得理解它们是怎么工作的以及它们的差别。

Why SOAP?

SOAP是成熟的协议，它有着完整、复杂的规范，它能够提供几乎所有的通信需要的解决方案。

它被设计用来暴露出Web Services中的各个操作。很多公司还在使用SOAP的一个原因就是它为组织中一些遗留的系统提供通信的支持。

更为重要的是以下几点：

  • 对于那些需要通过API和消费者的协议进行通信的应用程序，SOAP是一个很好的选择，因为它通过WSDL(Web Service Description Language)可以强制用正式的协议。
  • 安全性是SOAP另外一个强大的地方。WS-RM或者是Web Service Reliable Messaging允许SOAP在执行和异步处理消息的时候提升安全性。
  • SOAP允许带有状态的执行。SOAP被设计成在通信中支持状态的管理。

Why REST?

REST 是一个很容易理解的软件设计架构，基于HTTP协议还有它基本的维护方法，允许使用REST服务轻松编码还有记录应用程序。

  • REST可以更高效率的利用带宽，因为它比SOAP更加轻量级。和SOAP不同，REST不会保存状态，并且对于service的读取有缓存，这样可以提高性能还有可伸缩性。
  • REST专注于资源为基础的操作，继承了HTTP协议的操作(GET, PUT, UPDATE, DELETE, POST)。这让许多Web应用开发者与浏览器开发者都对REST感到很熟悉。
  • REST提供的简单以及便利是Google,Facebook,Amazon这些公司把它们的SOAP APIs转到RESTFUL APIs最强有力的原因。
  • REST被广泛用于一些需要大量往返消息的应用中，为了防止应用在某些情况下没有响应，REST允许激活其重新尝试这个过程。
  • REST允许通过URL进行简单而直接的调用。

分享一篇文章《正向代理与反向代理的区别》

文章介绍了什么是正向代理与反向代理以及它们之间的区别与联系。

正向代理

  • 位于客户端之间和目标服务器之间的代理服务器。
  • 正向代理需要你主动设置代理服务器ip或者域名进行访问，由设置的服务器ip或者域名去获取访问内容并返回。比如Google访问不了，通过购买一个VPS与SS来间接访问Google。
  • 它使得客户端对服务端不可见，隐藏了客户端的细节。

正向代理的使用场景

正向代理的典型用途是为在防火墙内的局域网客户端提供访问Internet的途径。

反向代理

  • 反向代理正好相反。对于客户端来说，反向代理就好像目标服务器。并且客户端不需要进行任何设置。
  • 反向代理是代理服务器端，为服务器收发请求，使真实服务器对客户端不可见。
  • 常见的Nginx就是一个很好的反向代理，常用来做负载均衡。

反向代理的使用场景

  • 保护和隐藏原始资源服务器
  • 加密和SSL加速
  • 负载均衡。比如常见的Nginx，它既可以做Http服务器也可以做反向代理服务器。
  • 缓存静态内容
  • 压缩
  • 减速上传
  • 安全
  • 外网发布
