Algorithm

本周算法题递增子序列

具体分析：

  1. 既然是找子序列，那么可以使用dfs进行回溯搜索。
  2. 当回溯时:
    1. 序列没有重复，需要搜索所有解空间。比如{1,2,3,4}，可以是{1,2},{1,2,3},{1,2,4};{2,3},{2,3,4};{3,4};{2,4}共7种。
    2. 序列有重复，当子序列的长度 >= 2的时候，以子序列开头的所有搜索空间都是确定的，因此只需要搜索一次。比如{4,6,7,7}这个集合为例，以{4,6,7}区间开头的递增子序列是确定的，因此回溯的第二个{4,6,7}可以不用搜索了。这样可以取到剪枝、去重的效果。
  3. 进入回溯的条件：
    1. 子序列为空
    2. 遍历序列的时候，子序列最后一个值小于当前遍历位置的值。
  4. 回溯的结束条件：
    1. 当已找到的序列已经被搜索过，那么这时可以直接返回。
    2. 所有的搜索解空间的操作都已经完成，这时也会返回，当然这时程序运行完毕就会自动做的事情。

Review

因为最近在学习数据库，因此本周阅读了一篇Medium上的文章《The Wide World of Databases》

作者介绍了几种常见的数据库系统以及它们的优缺点，然后给出了一些例子，给了解决现实生活中的问题。

关系型数据库

下面是一些常见的关系型数据库

  1. 关系型数据库是基于关系代数设计的数据库，你可以用关系代数在关系数据库里做任何事。
  2. SQL查询语言就是基于这些表达式去设计的。

优点

  1. 工业中经常使用
  2. 在建模和系统管理上非常有效率
  3. 非常适合简单的应用在明确定义的关键字中读取数据
  4. 有着AICD保证，提供原子性（atomicity，或称不可分割性）、一致性（consistency）、隔离性（isolation，又称独立性）、持久性（durability）。

缺点

  1. 一旦创建之后很难去更改它们
  2. 在很大的数量级中去查询的话会比较慢
  3. 数据库水平扩展的话不是内置的操作，因此都是从更高的一个抽象级别去扩展。现实世界中实现都是比较复杂的。

例子

  • MySQL
  • Postgres
  • SQLite
  • OracleDB

文档型数据库

文档型数据库一般都归在NoSQL的下面。它们一般都不遵循所谓的关系代数。一般和它们交互的话也不通过SQL。一般它们定义一个数据结构的话都比较灵活。一般数据库中的一行数据的话就是一个键对应一个二进制的JSON Object。您可以将文档分组为逻辑集合，其中对每个集合中的数据甚至相邻文档没有严格要求。

优点

  1. 非常简单的读写操作，就像查询操作一样。
  2. 水平扩展的话更为容易，因为数据与数据之间没有之间的联系。
  3. 存储简单，连续，独立的数据。

缺点

  1. 复杂的查询的话需要通过数据库和集合查询很多地方。
  2. 数据与数据之间很多的联系和链接。
  3. 很多实现都没有ACID保证。

例子

  • MongoDB
  • Couchbase
  • RethinkDB

键值存储数据库

键值类型的存储数据库经常用于一些简单的数据比如只需要给一个Key然后返回一个Value的这种情况。KV Stores可以用于磁盘上的存储持久化数据或者是存储临时数据用于提高效率或者性能。其实它们属于非关系型数据库，因为它们的设计也不遵循关系型数据库模式。

优点

  1. 很好的性能
  2. 扩展性很好
  3. 作为应用的支持性数据库的话能够很好的提高性能。

缺点

  1. 除了查找之外其他都很低效率
  2. 当进行复杂的查询的时候有很差的性能
  3. 内置的实现并不支持变为应用的主要数据库

例子

  • Redis
  • Memcached

列式数据库(Column-Oriented Databases)

列式数据库与传统的行式数据库不一样，它主要在一些特殊的场合使用。比如很大情况在online analytical processing (OLAP)上而不是传统的事务处理工作流上。

优点

  1. 适合大数据还有数据处理的应用
  2. 在很大的数据集合上做聚合
  3. 在查询和存储时间序列的数据的话十分有效。
  4. 系统中的数据处理好的话有很高的可扩展性

缺点

  1. 查询数据量小，分散的数据时候没有效率
  2. 对于Online transaction processing (OLTP) 这一种操作每场没有效率。
  3. 性能取决于key和data的布局与设计。

例子

  • Google Cloud Big Table
  • Amazon Redshift
  • Apache HBase

图类数据库

主要用于描述节点与节点之间联系的数据库，比如社交网络。当你发现你的数据可以用一系列相连的节点表示的时候，这个时候就可以用图类数据库。

优点

  1. 很容易表达数据与数据之间复杂的关系
  2. 很容易改变数据与数据之间的关系数据结构

缺点

  1. 只能查询基于关系的数据
  2. 对于OLAP应用查询和存储大量的数据很慢。
  3. 当没有优化好图的数据结构的时候运行算法会很慢。

例子

  • Neo4j

总结

作者介绍了几种数据库。不过个人认为学数据库的话先从关系型数据库开始学习，比如MySQL，然后了解一下数据库设计的几种范式与设计数据库需要注意的事情，然后就是多动手写SQL啦。

Tips

因为工作的原因，平时基本用不到数据库，所以决定学习一下数据库。于是这周复习了一下数据库，跟着慕课网上《数据库设计的那些事》学习了一遍。重新学习了一下数据库的几个基本范式，设计数据库需要注意的事项。下面需要重点记一下三大范式。

  • 第一范式（1NF）：强调的是列的原子性，即列不能够再分成其他几列。
  • 第二范式（2NF）：首先是1NF，另外包含两部分内容，一是表必须有一个主键；二是没有包含在主键中的列必须完全依赖于主键，而不能只依赖于主键的一部分。
  • 第三范式（3NF）：首先是 2NF，另外非主键列必须直接依赖于主键，不能存在传递依赖。即不能存在：非主键列 A 依赖于非主键列 B，非主键列 B 依赖于主键的情况。

编程的本质

分享一篇耗子叔在编程范式游记里面提到的《编程的本质》这篇文章。

引出观点

Program = Logic + Control + Data Structure

Niklaus Emil Wirth写了一本书《Algorithms + Data Structures = Programs》。这本书主要阐述了算法与数据结构的关系。

Rebert Kowalski发表了一篇论文Algorithm = Logic + Control》。这篇论文主要表达的是：任何算法都能分为两个部分，一个是Logic是用来解决实际问题的，一个是Control，这是用来决定用什么策略来解决问题。Logic部分是真正意义上的解决问题的算法，而Control部分只是影响解决这个问题的效率。程序的效率和逻辑其实是没有关系的，如果将Logic和Control分开，代码就会变得特别容易改进和维护。

通过分析这个表达式，可以得出：Program = Logic + Control + Data Structure

具体分析

其中Logic决定了算法的效率，而Control是决定用来描述如何使用逻辑。总结一下，所有语言或者范式都在表达几个东西。

  1. Control是可以被标准化的，比如遍历数组，查找数据，多线程，并发，异步等等，都是都可以被标准化的。
  2. 因为Control需要处理数据，所以标准化Control，需要标准化Data Structure，我们可以通过泛型编程来解决这个事情。
  3. 而Control还要处理用户的业务逻辑，即Logic。所以，我们可以通过标准化接口/协议来实现，我们的Control模式可以适用于任何的Logic。

上面就是泛型编程的本质。

  • 有效地分离Logic、Control和Data是写好程序的关键所在。
  • 有效地分离Logic、Control和Data是写好程序的关键所在。
  • 有效地分离Logic、Control和Data是写好程序的关键所在。

我们平时写代码，就会把很多业务逻辑和控制逻辑放在一起。里面有些和业务逻辑先关，有一些不相关，于是，这就导致了程序的复杂度。

耗子叔然后给出了一些逻辑与控制耦合度特别大的代码，然后给出了一个总结。

  1. 业务逻辑的复杂度决定了代码的复杂度
  2. 控制逻辑的复杂度 + 业务逻辑的复杂度 ==> 程序代码复杂
  3. 绝大多数程序复杂的原因：业务逻辑与控制逻辑耦合度大

有效策略

一些分离Control和Logic的策略。

  1. State Machine
    1. 状态定义
    2. 状态变迁条件
    3. 状态的action
  2. DSL - Domain Specific Language
    1. HTML, SQL, Linux Shell Script.
    2. AWK，正则表达式
  3. 编程范式
    1. 面向对象：委托、策略、桥接、修饰、IoC/DIP,MVC…
    2. 函数式编程：修饰、管道、拼接。
    3. 逻辑推导编程：ProLog

小结

没工作前如果让我来阅读这篇文章的话，是不会体会到这一篇文章有多棒的，只有在实际的写了几年代码，然后接触了MVP，MVC这些知识，才发现它们就是这一篇文章里面所阐述的内容。而面向对象，还有设计模式，高内聚，低耦合都是为了把Logic和Control分开，使得代码变得容易维护和开发才有的。
总的来说：

  • Logic才是真正有意义的(What)
  • Control部分只是影响Logic部分的效率(How)
