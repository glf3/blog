Review

本周阅读了一篇Medium上的文章《How Microservices Saved the Internet》

微服务的历史与介绍

　　作者说过去的话Web应用是部署以单机的形式部署的。如果工程师需要垂直扩容的话都是购买更多的服务器，这个成本高，而且会受摩尔定律的限制。

　　早期的水平扩容的话主要是部署重复的Web Server服务器。然而最近的微服务架构与云逐渐主导了游戏规则。微服务的概念是将一个很大的应用切分成一些独立的模块，它叫做服务，每一个服务完成具体的任务。与单一的Web Server从头到尾处理一个请求相比，微服务把应用拆分成几个Service，举个例子，可以拆分为：

  • 用户认证
  • 页面服务器
  • API 服务器
  • 数据库

　　每一个微服务由一个或者多个复制品（replicas）组成。对一个后端运行在Django上的网站而言，水平拆分的话可以将运行的Django服务器多复制一份，这些复制品就是replicas。如果我们在运行数据库的话，可以提高replicas的数量，通常我们把每一个数据库叫做shards。可以把shards看做一个单独的数据库，它独立负责全部数据库的一小部分。这些replicas可以运行在不同的机器上。如果我们把一个应用部署在100台机器上的话，微服务架构就是让我们可以把100个replicas部署在100台机器上面。

　　然后作者举了通过Google搜索的例子，来说明了Google搜索内部的微服务框架。

微服务的优点

  • 微服务最大的有点就是有能力进行任何组件的水平拆分。如果某一个服务过载了，你可以很轻易的拆分那个特定的服务。
  • 服务是独立的。每一个服务和组件可以被一个单独的工程师团队去维护和开发。每一个模块和服务可以是不同的语言开发的，比如数据库服务可能是用C语言开发的，而Web Server可能是Python开发的。这一些服务可以被更新然后新的模块也可以被实现而不影响现在的服务流程。
  • 架构是高度可配的。一个公司不需要开发所有的服务与模块。第三方应用像MySQL和Redis可以很轻易的集成进去。
  • 容错机制。一个服务的replica失败的话不会导致整个service失败。一个特定服务的bug不会影响与它不相关的服务。这个一点也不简单，容错机制在分布式数据库和网络中一直都是一个有趣的话题。

微服务的缺点

  • 有着延迟大，复制和共享数据方面的问题
  • 部署和debug的难度是随着微服务的容量增大而增大的

Tips

这周复习了一下Android中App进程的启动过程。

当我们点击桌面Launcher的app icon的时候，如果进程没有启动的话会执行以下操作。

  • Launcher调用acitivity.startActivity
  • startActivity通过Binder Call给AMS发送启动activity的消息
  • AMS判断待启动的应用的进程是否启动，如果没有启动进程，则进入到启动应用进程的流程
  • 如果启动的进程，则进入启动activity的流程。

启动进程的流程：

  • AMS通过Socket通信和Zygote进行通信，把需要传递的进程的信息传递给Zygote
  • Zygote在其监听的/dev/socket/zygote发现相应的请求之后，会fork自身，并且返回相应的PID
  • 这个Process会初始化，并且启动Binder线程池，使得它具有IPC通信的能力。
  • 调用ActivityThread这个类的main方法，开启Looper，打开主线程的消息循环机制。到此为止整个应用启动完毕。

进程启动的时机

  • 启动任意Activity、Service、BroadcastReceiver、ContentProvider中的一个任意一个，AMS都会判断进程是否启动，如果没有启动的话，是需要先启动进程的。

其他

  • 在AndroidManifest里面对四大组件声明android:process=”processName”指定四大组件运行的进程，默认组件运行进程名和包名一致。
