  3. 当lower_bound的值 index >= 0 && index < intervals.length的时候，代表找到了合法的最右区间。

优化方式

使用索引

在MySQL中，有这么几种数据可以用来作为索引。

  • (Primary Key)主键：在MySQL中，主键默认就已经被建成索引了。
  • (Unique)：独一无二的值。
  • (Index)：除了主键之外其他的域或者属性。
  • (Full Text)：一整个字符串。

索引的储存有两种方式：

  • Hash(哈希的方式)，这个适用于绝对匹配，但不适合用于比较。
  • B-Tree(二叉平衡树)，这是一种很常见的方式，用它来存储索引是十分快捷与方便的。

Partitioning(分表)

　　当我们建立索引的时候，我们只有关于那个建立的索引的属性的信息。我们没有关于其他不在索引里面属性的信息。所以，有一些时候MySQL需要往主数据库表里面去查找其他的属性。
　　因此，这个时候就可以使用分表这一项技术，即把主表里面的数据通过切割的形式，分成几个表，但在管理和使用上面，还是作为一个去管理和操作。当数据被拆分的时候，MySQL会有一个映射，这个映射可以让我们根据需要的Key去找响应的表。

　　但是，分表的话也是在同一台机器上。

Partitioning(分片技术)

　　在一个大型的系统中，把所有的数据存储在一台机器上是十分麻烦的。

　　一个特定的表是很重的并且承载了大部分的查询任务，而另外的表可能查询的任务没有那么中，但是互相它们又有着联系，不能随意的切割。

　　换句话说，最近的三个月的数据是经常用的，而老一点的数据就不经常使用。

　　为了解决这个问题，我们可以把最近三个月的机器放到另外一台机器上去。分片是把大数据切为很多小的碎片，然后把它放在不同的关系数据库中去。换一句话说，分片是水平的切分技术。

总结

　　作者通过一个简单的例子层层深入，给出了优化关系型数据库的方法。关系型数据库有着能力和应用一起成长，因此使用好关系型数据库是十分必要的。
其中关于优化数据库操作的话有下面几种方式

  • 建立对应的索引
  • 分表：适用于表很多导致数据很多的情况，它把一个主表拆分分为很多其他表。它属于垂直切分。
  • 分页：适用于表少，但是数据量大的情况。它把数据切分放入放入不同的机器中、数据库中。它属于水平切分。

Tips

　　最近花了一周左右的时间阅读了《图解HTTP》这本书，系统地了解了关于HTTP的知识。包括HTTP请求方法，请求头部，响应头部，返回状态码还有Cookie与Session，HTTP与HTTPS的区别，我觉得HTTP与HTTPS的区别需要重点记一下。

  1. HTTPS = HTTP + 加密传输（SSL和TSL） + 证书 + 完整性验证。其中证书的话得去权威的第三方认证机构(CA)购买。
  2. HTTP加了SSL(Security Socket Layer)和TSL（Trasport Layer Security）即是HTTPS。通常HTTP是直接与TCP通信，使用HTTPS之后，则是先和SSL进行通信，在和TCP进行通信。
  3. HTTPS使用的加密的方式有两种，第一种是公开密匙加密，第二种是共享密匙加密，使用HTTPS时一般两种加密方式都有使用，属于混合加密。
  4. 一般非敏感信息通信时使用的是HTTP,敏感信息则使用HTTPS,因为HTTPS使用SSL进行通信的话，SSL传输数据比HTTP直接和TCP传输数据慢很多，而且涉及到加密、解密，这样的话对服务器的资源是一个不小的挑战，特别是访问量较多的Web网站。

　　下面就来介绍一下Spring的设计目标以及整体架构。

Spring的设计理念

  1. Spring为开发者提供的是一个轻量级应用开发框架，作为平台，Spring抽象了我们许多应用开发中遇到的问题。
  2. Spring支持POJO和使用JavaBean的开发方式，使应用面向接口开发，它充分支持OO(面向对象)的设计方法。

比如在传统的EJB开发需要依赖按照J2EE的规范实现的J2EE应用。在设计，特别是实现的时候，需要遵循一系列的借口标准和规范，才能在服务器的环境中测试和部署。

　　而Spring使用了相对EJB而言轻量级的开发思想，即使用POJO的开发方式，是需要使用简单的Java对象或者JavaBeans就能进行Java EE的开发，这样开发、测试、部署都得到了简化。

　　Spring的设计核心有以下两点。

  1. 使用IoC容器可以轻松的管理对象之间的耦合关系，可以通过外部的一个或几个XML文件，可以方便的对应用对象之间的耦合关系进行浏览、修改和维护。
  2. 通过AOP(面向切面编程)，可以动态的以非侵入式的方式来增强服务的功能。

实际上可以把IoC和AOP看做是Spring的Kernel层。通过IoC和AOP以核心构建一些服务的抽象和应用，它们本身的松耦合设计理念，应用可以简单的开发接口或现成的应用模板，就可以方便的使用Java EE的服务。

　　对用户而言，绑定的是IoC容器和AOP模块，而不是具体的Java EE服务。这样可以让用户选择不同的服务实现提供了基础。比如，根据应用需求，用户可以选择使用Hibernate作为ORM工具，也可以使用iBatis，还可以使用其他类似的工具。这也体现了Spring的设计理念，面向接口编程而不是面向实现编程。

　　总的来说，Spring的设计理念是让基于Java EE的应用开发的编写变得更加简单与高效，让用户专注业务逻辑的开发，让可移植性变得更好。

Spring的整体架构

　　Spring的架构可以分为三个模块：核心、组件、应用。

  1. 核心：IoC容器，AOP核心模块。
  2. 封装的Java EE服务，作为中间的驱动组件。
  3. 其他作为上层的应用

其中核心是：IoC容器，AOP核心模块。

中间驱动组件：事务处理，Web MVC，JDBC，ORM，远端调用等。

上层应用：这些应用包括来源于社区的应用封装，也包括Spring作为平台开发出来的各种类型的应用。

Spring的应用场景

　　在Java EE企业应用开发中，最基本的Spring的使用场景就是大家熟悉的SSH架构来完成企业级的开发，从而取代传统的EJB开发模式。

　　在SSH架构中，Struts作为Web UI层、Spring作为中间件平台，Hibernate作为数据持久化工具（ORM）来操作关系数据库。比如使用Apache Tomcat，MySQL数据库和Linux环境，这就是一个完整的使用开源软件搭建企业应用的典型案例。

总结

  1. Spring是一个非侵入式的框架，其目标是使应用程序对框架的依赖最小化，应用代码可以在没有Spring或者其他容器的情况下运行。
  2. Spring提供了一个一致编程的模型，使得应用可以直接使用POJO开发，从而和运行环境（如应用服务器）隔离开来。
  3. Spring推动应用的设计风格向面向对象以及面向接口编程转变，提高了代码的可重用性和可测试性。
  4. Spring改变了体系架构的选择，虽然作为应用平台，Spring可以帮我们选择不同的技术实现，比如从Hiberante切换到其他ORM工具，从Struts切换到Spring MVC，尽管我们通常不这样做，Spring给我们提供了选择技术方案的可能性，降低了平台锁定的风险。
