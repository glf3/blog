Review

本周阅读了一篇Medium上的文章《How to run Docker and get more sleep than I did》

　　作者在一家名为Gusto的薪酬计算公司工作，该公司的工作主要是用于抵押付款，学分制收费，还有子女抚养费方面的工作。由于工作的性质要求了程序了实时性，准确性。所以每一个提交都需要经过通过Ruby, Python, and Go, 经过成百上千次的运行来测试每一个提交。

　　他们通过很重量级的Docker化的持续集成流水线(heavily Dockerized continuous integration (CI) pipiline)
来完成这个工作。

　　这篇文章就介绍作者在使用Docker进行持续继承的过程中遇到了一个奇怪的bug，并且介绍了一些关于使用Docker的最佳实践。

　　而这些最佳实践，都是作者花了成百上千个小时查看Docker日志，修复Docker奔溃，还有定位Bug所在的代码总结而成的最佳实践。

　　作者通过分析一个线上的bug，然后在本地复现了这个bug，分析这个bug导致的原因是Ruby的不正确使用导致在后台给Docker太大的压力，引出了一个Docker的bug，导致Docker hang住了。然后给读者介绍了Docker的基础概念，Docker能做什么，以及Docker使用中的一些最佳实践。

Docker提供了一种定义，实例化和维护隔离执行环境的机制。它比进程更重，比虚拟机更轻。

Docker中的名词

  • Image（镜像）：不可变的二进制文件存放在磁盘中或者是Dockerhub中，它可以被实例化成容器。docker images
  • Container(容器)：一个实例化的容器，它可以执行任意命令。docker container ps
  • Host System(宿主系统)：运行容器的底层系统。如果你在笔记本上使用docker的话，那么笔记本就是你的宿主系统。
  • Volume(数据卷)：一个供容器可以访问的文件系统。容器与容器之间可以共享数据卷。
  • docker-compose(一种给用于构造容器的编排工具。通过提供.yml产生Docker命令，总之，可以用docker-compose做的，都可以用docker做)

Versioning(版本控制)

  • 当从一个基础镜像继承而来的时候，需要明确的指出基础吉祥的版本，比如(FROM:ruby)，应该变为(FROM:ruby:2.3.4)
  • 在docker-compose.yml文件里面，永远明确通过version key明确制定docker-compose的版本
  • 注意定期更新Docker的版本，有时候Docker binary是处在向后兼容的模式。

Layering(分层)

Layers是一些存储在宿主操作系统上的简单的文件，它们可以通过Docker的文件系统驱动去生成最终的镜像。正确使用layer的话可以大大简化发布和debugging的时间。当然，对layer的不熟悉的操作大大增加了没必要的工作。

  • Dockerfile里的每一行最终都生成了一个”layer”。
  • 每一个layer都取决于之前layer。每一个layer都做了缓存，每一个Docker镜像文件都代表着一系列的layer。
  • 低频率的操作的命令应该尽早出现在Dockerfile中，与此相反，高频率的操作应该写在文件的底下。这会加速Docker的构建过程。
  • 如果你发现拷贝了很多相同的代码在两个不同的Dockerfiles中，这时可以考虑使用multi-stage（多文件构建）。

Isolation(隔离)

  • 当使用docker-compose的时候，使用参数–project-name去构建（为容器生成前缀），参数可以是一些独特的值，比如BUILDKITE_JOB_ID,或一些派生数字。
  • 当docker-compose结束之后，显示的使用docker-compose down或者是docker stop
  • 当你必须手动去构建容器的时候，声明数据卷和特定的网络，从而避免了cross-run这种情况。

Resource Utilization(资源利用率)

  • 单个的容器是没有资源限制的，它的瓶颈就是kernrl/host操作系统给它的资源。
  • docker还有docker-compose支持一些特定的标示，从而可以限制一个容器使用的内存，CPU，还有swap
  • 使用 docker system prune –volumes阶段性的清理我们的镜像，volumes，还有网络。这一些与当前运行的容器没有关联的资源。为了避免一些极端的情况，可以间断性的清理资源，比如可以使用docker container prune –filter “until=24h”

(Deploying)发布

Dockerhub写的话经常失败，它很可能随机的导致覆盖一些镜像。

  • 我们自己维护了一个开源Docker Registry通过write-around cache的方式消除了对于Dockerhub的读依赖。
  • 使用 image_name:latest去获取最新的镜像是一种anti-parrern的方式，这样旧的镜像文件会被覆盖。
  • 确保你的镜像在Dockerhub上永远都有一份备份。比如gusto/zenpayroll:latest, 然后针对它使用MD5/SHA打一个标签，可以是gusto/zenpayroll:abc123.
  • 把Docker镜像和layers当做文件存在文件系统里面，然后可以通过引导脚本使用它们。

Volumes(数据卷)

  • 当你在Docker里面开发的时候，在宿主文件系统使用volume in替代ADD‘ing和COPY’ing操作，这个可以大幅降低迭代周期。
  • 不要把个人的秘密资料放进镜像文件里面。
  • 在宿主文件系统里面，root拥有的文件是很容易被删除的。所以尽可能的在非root的情况下，使用容器操作进行读取磁盘的操作。
  • 使用数据卷的时候，首要把数据卷命名为匿名的。
  • 当你在宿主容器中使用volume in文件的时候，这会引入相同的UIDS/GIDS。所以对待ADD指令需要非常谨慎。

杂项奇怪的问题

  • Docker有时候会hang起来，当在一行中输出很多东西到STDOUT的时候，这是一个公认的很常见的Docker的bug。
  • 为了避免这个问题，可以把一行个很长的文字，分行输入到STDOUT中去。

Debugging Tips(Debug的建议)

  • 确保一些基本的工具存在Dockerfile中，比如，vim/nano,curl等。
  • 利用图层与FROM。比如一个作为Dockerfile最宝贵的一部分可以使用apt-get install MyCustomPackages && MoreCustomPackages的话。如果可以把这个放进自己的layers里面的话，那么接下来就可以复用它。
  • 有时候，如果你发现不知道有一些东西为什么引入的话，那么可以把其他的layers都排除掉，除了那一个你认为可能导致问题的layer，然后在shell中执行它，并且观察安装环境。
  • docker exex -it 允许让你进入一个存在的容器。使用docker exex -it 命令去debugging一些破损的容器。
  • docker ps可以打印出所有正在运营的容器。然后使用docker exex -it 命令去debug。
  • 使用参数–verbose。它可以解决大部分粗糙的构建问题。
  • 耐心的去查看很长的出现问题的日志。

总结一下

作者通过一个debug的经历，引入了Docker。介绍了Docker的基本概念还有Docker的一些最佳实践。对于我一个这样没有使用过Docker的人来说，理解起来还是比较困难的，接下来的话时间得学习下Docker的主要用来解决什么问题的；还有Docker的一些基本使用与概念。

Tips

记一次开启BBR更新Ubuntu14.04内核遇到的问题与踩过的坑

　　响应耗子叔的号召，趁着美国那边的黑色星期五购买了一个VPS，然后准备使用VPS+BBR。家里的电脑使用的是Ubuntu14.04，内核是版本是4.2，BBR需要内核版本 >= 4.9。于是需要更新内核。下面记录一下更新中才过的一些坑与遇到的问题。

当更新Ubuntu的内核的时候，最好去官网下载对应的内核版本。当安装了最新的内核之后，最好不要把老的内核删除。

第一个问题

　　当时我在网上随便找了一个内核4.12的版本，然后使用sudo dpkg -i *.deb更新内核，全程没有报错，但重启机器之后，出现了循环登录的情况。后来发现是显卡的NVIDIA驱动与最新的内核不兼容，需要更新驱动。由于进不去Desktop，于是我把新版本的内核卸掉，继续使用旧版本的内核。这里着重提一句，当我们安装上了新版本的内核的时候，不要随便的把旧版本的内核删除，有可能新版本的内核不生效，这样我们还可以继续使用旧版本的内核。

第二个问题

更新NVIDIA显卡驱动不要通过apt-get install去下载，需要去NVIDIA的官方网站下载最新的显卡驱动，然后本地编译。

　　这个时候卸载的最新的内核，然后进入桌面，通过apt-get install NVIDIA的官方驱动，然后继续安装上新的内核，重启。
重启之后发现还是进不去桌面，这个时候发现可能是NVIDIA驱动不对导致的，于是卸载了最新的NVIDIA驱动，卸载了最新的内核。老老实实的去NVIDIA的官网下载与电脑显卡对应的内核，我的显卡是NVIDIA的10系列，对应的最新驱动版本是410.78。于是从官网下载到本地。安装新的内核，然后本地安装NVIDIA驱动，这时需要把X Server关闭，并且关闭集成显卡驱动。然后重启电脑。

第三个问题

安装过程中注意查看安装日志，因为有的时候可能显示正常安装了，通过命令查看也安装成功，但这个时候可能还会有一些问题出现，这个时候就要注意有没有报错日志了。

　　重启之后发现还是进不去桌面，于是又走了装显卡驱动的过程。在过程中发现安装有报错，发错是发现找不到对应的一个文件linux-headers，后来才发现原来我安装的新的内核文件的它不全，还差了一个.deb包，所以又去官网下了全部内核文件进行编译。
有了之前的教训，这个时候就会注意查看安装日志，安装内核的时候发现有报错，细看一下原来是内核编译的gcc必须大于5.0才行，于是又安装了gcc-5 g++5，这个可以通过apt-get install去安装。安装了新的编译环境并且更新内核之后重启电脑。发现还是进不去桌面，这时又准备重新安装NVIDIA显卡驱动，在安装显卡驱动的过程中发现有报错，说内核的编译环境是gcc-5.3.6而NVIDIA的环境是需要的环境是gcc-6.0+，于是这个时候，就把gcc更新到6.0+，然后重新编译NVIDIA显卡驱动。

　　在安装的过程中，NVIDIA显卡驱动会提示要不要添加DKMS，第一次选择了添加，然后它提示当前的gcc版本与DKMS需要的不符合，是否继续安装，当时选择安装，后来发现选择之后NVIDIA驱动并没有更新成功。于是再一次安装NVIDIA驱动的时候选择不需要DKMS，这时候它会提示当前的编译环境是gcc-6.3.4，与NVIDIA需要的编译环境6.2.3不一致，是否继续，我想了一下gcc编译环境版本差别不大，应该不会有什么问题，于是选择继续，之后再进桌面就发现成功了。使用uname -r 查看内核还有nvidia-smi查看显卡版本，都是正确的了。

　　之后我们就可以开心地开启BBR使用VPS了。

总结

  • 下载驱动的更新的话去尽量去官网上下载。
  • 遇到问题尽量不要先去Google，先查看有没有具体的报错日志，然后针对性的去搜索，解决。
  • 要有耐心。

一些零散的思考与看法

人文知识的学习

我觉得理科生得具备一定的人文知识。平时可以把80%的精力放在专业的学习上，20%时间花在其他书籍的阅读，比如哲学，文学，以及历史都行。

　　这样我们在除了知道自己所学专业的知识，也能够享受人文情趣。当我们未来对专业之外的领域感兴趣的时候，也可以迅速的学习其他领域的知识。

开放包容的心态

　　从事客户端工作的同学可以学习一些和前端、服务端的知识，服务端的同学可以学一些客户端和前端的知识，这样可以让自己的技术栈和视野都变得开阔。

　　多交朋友，老的少的都行，说不定可以改变自己的一些过去的看法与观念。

　　当我们具有包容的心态，持续学习的能力以及深厚的专业知识，持久的阅读习惯的时候，未来的道路也会越走越宽。
